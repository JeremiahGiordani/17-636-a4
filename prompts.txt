STEP 1 (BASE SYSTEM) -- STAGE: SPECIFY
================================================================
We are building a small web app for a course assignment: a MINIMAL SPREADSHEET
(a stripped-down Google Sheets). A grid of cells; each cell holds either a
literal value or a formula starting with "=" (e.g. =A1+B2*3). Formulas can
reference other cells (like A1). Editing a cell recalculates all cells that
depend on it. Formula evaluation happens on a Python/Flask BACKEND (this is the
interactive backend); the frontend is a simple HTML/CSS/JS grid.

This is the SPECIFY stage. Write a clear, concise specification to a file named
SPEC.md at the repo root. Do NOT write any application code yet -- only SPEC.md.

The spec must cover:
- A one-paragraph overview.
- IN SCOPE for the BASE system: an N x N grid (say 10x10), literal numeric
  values, formulas using + - * / and parentheses, cell references like A1, and
  automatic recalculation of dependents when a cell changes. The Flask backend
  evaluates formulas.
- OUT OF SCOPE for the base (explicitly deferred to a later EXTENSION step):
  functions like SUM/AVG/IF, range references like A1:A5, saving/loading sheets
  to a database, styling/formatting, and multiple sheets.
- Tech stack: Python 3 + Flask backend, vanilla HTML/CSS/JS frontend, pytest for
  tests. (SQLite only later, for the extension.)
- The backend API: endpoint(s), request and response JSON shapes.
- Formula/edge-case behavior: a reference to an empty cell counts as 0; division
  by zero yields an error value; an invalid/non-numeric formula yields an error;
  circular references are detected and reported as an error instead of looping.
- A concrete ACCEPTANCE TEST LIST: a numbered list of specific, testable cases
  written as (input cells -> expected computed output), covering arithmetic and
  precedence, parentheses, cell references, recalculation after an edit,
  empty-cell references, division by zero, and circular-reference detection.
  These cases will become pytest tests in a later stage.

Keep it unambiguous and not too long. Write only SPEC.md.

STEP 1 (BASE SYSTEM) -- STAGE: PLAN
================================================================
This is the PLAN stage for the BASE system. Read SPEC.md, then do two things.

(1) Write PLAN.md at the repo root, concise, containing:
- The file/directory layout: a Flask app (app.py), a formula-engine module
  (engine.py), templates/index.html, static/ for CSS+JS, and tests/.
- The shared interfaces both the tests and the app code will use: the formula
  engine's public function(s) with exact signatures and return shapes, and the
  Flask endpoints (GET /, GET /cells, POST /cells) matching SPEC.md.
- An ordered task breakdown for the build stage.
- A note that the pytest suite in tests/ is the DEFINITION OF DONE: the build
  stage must make the code pass it without weakening the tests.

(2) Write the pytest TEST SUITE encoding SPEC.md's 16 acceptance tests, under
tests/ (e.g. tests/test_engine.py for the formula engine and tests/test_api.py
for the Flask endpoints via Flask's test client), using the interfaces you
defined in PLAN.md. Add a conftest.py or pytest config so tests import the
app/engine modules from the repo root.

Do NOT implement any application logic yet -- no engine evaluation code, no
Flask routes, no frontend. Only PLAN.md, the test files, and test-support config.
It is expected and fine that the tests currently fail because the app code does
not exist yet. Keep PLAN.md short and clear.

================================================================
STEP 1 (BASE SYSTEM) -- STAGE: BUILD  (fed to claude by loop.sh each iteration)
================================================================
You are building the BASE system of a minimal spreadsheet web app. This is ONE
iteration of a loop; the repo is your memory across iterations.

Each run, do this:

1. Read SPEC.md, PLAN.md, and progress.md to understand the goal and what is
   left. The pytest suite in tests/ is the definition of done.
2. Run `python3 -m pytest -q` to see what is currently failing.
3. Implement or fix the NEXT chunk of application code to make failing tests
   pass, following the interfaces in PLAN.md. Order: formula engine (engine.py)
   first, then the Flask app (app.py) with GET / , GET /cells, POST /cells,
   then the frontend (templates/index.html + static/) so the 10x10 grid works
   in a browser. A Python virtualenv is active (.venv) — use it; if you add a
   dependency, pip-install it and add it to requirements.txt.
4. Do NOT weaken, delete, or trivially rewrite the tests in tests/ to make them
   pass — make the code correct instead. (Only fix a test if it clearly
   contradicts SPEC.md, and say why in the commit.)
5. Run `python3 -m pytest -q` again to check progress.
6. Commit your changes with a clear message, e.g. "base build: implement
   formula engine".
7. Update progress.md: check off what is done, list what remains.
8. If ALL tests pass, confirm app.py serves the grid page, mark the base system
   complete in progress.md, and commit.

Work in focused increments — one meaningful chunk per run. Keep going until the
entire test suite passes.
